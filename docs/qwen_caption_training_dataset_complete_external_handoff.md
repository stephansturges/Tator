# Qwen Caption Training Dataset Complete External Handoff

Date: 2026-06-29

## Purpose

This document is the concise external handoff for the caption-derived VLM
training-dataset work. It explains what was implemented, why the workflow is
structured this way, what artifacts it produces, how the implementation protects
source truth from generated language, what has been hardened, and what still
must be proven before a generated corpus is used for fine-tuning.

The document is intentionally dataset-neutral. Do not add customer, partner,
dataset, or project code names before sharing it. The implementation and review
contract do not require those names.

## Executive Summary

The caption panel now supports two distinct workflows:

- **Captioning for annotation assistance**: generate, store, compare, promote,
  and export image captions.
- **VLM training-dataset creation**: generate auditable image/question/answer
  rows from a dataset, preserve provenance, support human review, and export
  trainer-ready JSONL only after validation gates pass.

The training-dataset workflow is exposed as **Create VLM training dataset**. It
can produce:

- one broad per-image caption row, called `caption0`;
- configurable VLM-generated visual question/answer candidates per image;
- optional deterministic metadata QA rows generated from trusted labels;
- flat trainer JSONL;
- per-image construction archive JSONL;
- candidate-level review JSONL;
- run-level instruction report JSON.

The key product decision is strict evidence separation. Source labels remain
trusted structured evidence. Captions and generated QA remain generated
language. Deterministic metadata QA is code-generated from trusted labels. Flat
trainer rows are selected exports, not the source of truth.

The current implementation is ready for external implementation review and a
small real-data pilot. It is not a claim that every generated corpus is already
training-grade. A corpus should be used for fine-tuning only after a real-data
pilot, human review where required, reviewed-row import, strict re-export, and
trainer-loader or fine-tuning dry-run validation.

## What Was Done

### 1. Added A Separate Training-Dataset Workflow

The UI now has a dedicated instruction-dataset section rather than hiding
training-data generation behind ordinary caption export.

Implemented controls:

- **Create VLM training dataset**
- **Generated QA per image**
- **Generated QA mix**
- **Generated answer format**
- **Include caption0**
- **Include generated QA**
- **Include deterministic metadata QA**
- **Give generator read-only label context**
- **Strict QA grounding**
- **Require ready report for trainer JSONL**
- **Download instruction JSONL**
- **Download instruction archive**
- **Download review JSONL**
- **Import reviewed JSONL**
- **Download instruction report**

The UI refuses to launch a training-dataset job if all row families are
disabled. Generated QA can be configured as archive/review-only, and the
confirmation text distinguishes those candidates from rows that will be
flattened into trainer JSONL.

### 2. Added Caption0 Plus Multi-Prompt QA

`caption0` is the broad scene-description row for an image. Generated visual QA
rows are additional VLM-created question/answer candidates over the same image.

This was necessary because a single caption per image is usually too thin for
VLM fine-tuning. Multiple prompts let the corpus cover broad descriptions,
object presence, counts, spatial relationships, attributes, and shorter
grounded answers.

Generated QA rows are not trusted source labels. They are parsed, structurally
validated, deduplicated, archived, and then either selected for trainer output
or rejected with explicit reasons.

### 3. Added Optional Deterministic Metadata QA

Deterministic metadata QA is optional and off by default. It is generated by
code from trusted labels, not by the VLM.

Supported deterministic categories include:

- visible class list;
- object-count schema;
- per-class counts;
- positive class presence;
- negative class presence;
- simple source-geometry spatial facts when supported.

This option exists because label-derived QA can be valuable training signal,
but it changes the corpus mix. Keeping it explicit prevents reviewers from
mistaking code-generated label facts for generated visual-language rows.

### 4. Added Four Artifact Families

The workflow emits four artifact types from the same run.

`instruction_training_rows` is flat trainer JSONL:

```json
{
  "image_path": "images/example.jpg",
  "question": "Describe this image in detail.",
  "answer": "A grounded answer.",
  "metadata": {
    "qa_id": "stable-row-id",
    "row_type": "generated_qa",
    "answer_source": "generated_qa_record",
    "answer_format": "natural",
    "validation_status": "accepted",
    "review_status": "accepted",
    "source_archive": "tator_caption_instruction_archive_v1"
  }
}
```

`instruction_archive_rows` are one JSONL record per image. They preserve source
annotations, `caption0`, generated QA candidates, deterministic metadata QA,
rejected rows, selected flattened rows, and consistency proof metadata.

`instruction_review_rows` are candidate-level review rows. They expose stable
QA ids, image path, row origin, question, candidate answer, selected training
answer, source summary, validation status, review status, rejection reasons,
and blank review-decision fields.

`instruction_report` is the run-level readiness and quality report. It records
counts, row-type distribution, split metrics, rejection reasons, source-field
provenance, artifact consistency, export validation, and training readiness.

### 5. Added A Closed Human Review Loop

The operator can export review JSONL, review generated-language candidates
outside the app, import accepted/rejected/needs-revision decisions, and then
re-export trainer JSONL.

Review import is metadata-only. It can apply review decisions and notes to
matching `caption0` and generated-QA records. It does not edit source labels,
boxes, image paths, generated questions, generated answers, deterministic QA,
or final annotations.

This keeps the training-data workflow auditable. A reviewer can decide whether
a row is trainable without silently rewriting source truth or generated
candidates.

### 6. Added Browser, Backend, And Trainer Validation

The implementation intentionally validates the same artifact family at multiple
layers:

- the browser validates rows and artifact consistency before downloads;
- the backend validates exports and review imports for API/script callers;
- the trainer loader validates flat rows before fine-tuning starts.

The trainer loader is the last safety boundary. It imports flat
image/question/answer rows, preserves metadata, resolves image paths, and
rejects rows with missing provenance, unknown validation states, rejected or
needs-revision review states, invalid JSON answers, duplicate canonical
image/question pairs, or unresolved image aliases.

## Why It Was Done This Way

### A Training File Needs More Than A Caption

Caption-only JSONL is useful for annotation assistance but weak for fine-tuning.
A model benefits from multiple views of the same image: broad captioning,
object-level questions, spatial questions, count questions, and concise
answers. The new workflow therefore supports `caption0`, generated QA, and
optional deterministic metadata QA.

### Generated Language Must Not Become Source Truth

Generated captions and generated QA can be wrong. Source labels are the trusted
structured evidence. The archive keeps those layers separate:

| Layer | Purpose | Written by VLM? | Direct trainer input? |
| --- | --- | --- | --- |
| `source_annotations` | trusted label facts and deterministic geometry | No | No |
| `language_annotations` | `caption0` and generated QA candidates | Yes | Only after flattening |
| `deterministic_metadata_qa_pairs` | code-generated label QA | No | Only when enabled |
| `flattened_training_rows` | selected model-visible rows | Derived | Yes |

When generated language makes a structured claim that can be checked against
source labels, the flattened training answer can be rewritten from trusted
source annotations. If the claim cannot be checked or contradicts trusted
labels, the row remains in the archive/review file but is rejected from trainer
JSONL.

### Flat Trainer Rows Need Rich Audit Artifacts Beside Them

The trainer needs a simple `image_path`, `question`, `answer` row shape. That
shape is correct for training but too thin for audit. The archive, review file,
and report preserve the evidence needed to explain why each row exists and why
it was accepted or rejected.

### Long VLM Jobs Need Set-And-Forget Semantics

Dataset-scale generation cannot depend on a browser tab staying alive or a
single model call always behaving well. The durable path uses backend jobs,
progress polling, attach/recover behavior, persisted status, loop detection,
recovery telemetry, and artifact gates.

Direct single-image captioning remains useful for diagnostics, but the default
dataset-scale posture is set-and-forget.

## Source Annotation Behavior

Source annotation summaries are built from real label evidence and deterministic
geometry. They may include:

- object counts;
- visible classes;
- annotation summaries;
- bounding-box instances;
- coarse geometry;
- simple deterministic spatial facts;
- uncertainty records;
- field provenance.

Important behavior:

- A missing label file is recorded as missing source evidence.
- An empty label file is treated as a real no-source-objects condition when the
  annotation workflow defines empty files that way.
- The VLM is not asked to create label-specific facts for objects absent from
  trusted labels.
- Images with no trusted labels can still receive visual captions and generated
  visual QA, but source-grounded count/class/spatial claims cannot be flattened
  unless source annotations support them.

For external sample packets, image payloads should be packaged with the review
artifacts rather than referenced only by an absolute local path. A review packet
should include the same images, labels, trainer JSONL, archive JSONL, review
JSONL, and report JSON from one run so the reviewer can reproduce image-row
resolution without relying on a mutable local image root.

## Runtime Hardening

### Output Token Budgets

The UI now separates Auto output-token behavior from explicit numeric override.
Thinking-capable models can receive higher automatic budgets because reasoning
traces can consume far more generated tokens before the final answer. A numeric
user value remains a hard per-call cap after schema clamping.

Prompt preview and runtime traces report the effective output-token budget so
operators can reconcile UI settings with actual model calls.

### Prompt Size And Dense Boxes

Dense scenes no longer require dumping every label box into a single prompt.
Counts remain complete and authoritative. Box lists become representative
spatial subsets when needed for prompt size, selected for class and region
coverage. Prompt wording should make clear that omitted boxes are not absent
objects.

Prompt-size accounting estimates rendered prompt pressure and can reduce
automatic output budgets when the prompt is already large. Explicit user caps
are not silently raised.

### Loop Detection And Recovery

Streaming generation now acts as a live repeated-output inspector. Repeated
punctuation or repeated-token loops are detected while tokens arrive. The
stream is closed, the runtime can be unloaded, and recovery routes through the
configured safe path. A trimmed repeated fragment is not accepted as a
successful caption.

Recovery can use safer decoding, fallback models, text-only composition from
completed crop observations, or deterministic count/layout fallback when
authoritative counts exist. Deterministic fallback is explicitly degraded
recovery, not the normal product path.

### Model Availability

The model dropdown visually distinguishes local models from models that require
download. Download-needed options are red; local options use the normal local
color. Backend jobs fail preflight for missing models unless downloads are
explicitly allowed.

### Full-Image Composition

The full-image composition stage can feel slower because it combines full image
evidence, crop observations, authoritative counts, representative spatial
context, and the user caption policy. The model is not intentionally reloaded
only because the pipeline reaches this stage. Reloads are expected when a
different model/runtime is selected, when the model is not kept resident, or
when loop/error recovery deliberately unloads the runtime before retrying.

## UI And Archive-Mutation Hardening

Caption and instruction jobs mutate the caption archive. During active archive
mutation, the UI disables or action-time guards:

- caption image, batch caption, caption-all, and training-dataset launch;
- ordinary caption exports;
- instruction trainer/archive/review/report downloads;
- reviewed JSONL import;
- full dataset ZIP download;
- caption text editing and autosave;
- manual caption add/update/primary/delete controls;
- alternate selection controls;
- prompt stack editors;
- style and opening text;
- glossary edit/reset/save;
- model choices;
- token, box, windowing, decode, health, pilot, and set-and-forget settings;
- caption dataset selection and refresh where the change would make the active
  panel display a different archive context.
- current-image caption archive reloads, unless the reload is the explicit
  handoff from a completed caption job.

The same principles exist on backend routes. API/script callers receive busy
responses before reading or mutating archive-backed data when a caption dataset
job owns the same dataset. Single-image and batch caption archive reads return
`caption_read_busy`; single-image and batch text-label mirror reads also return
`caption_read_busy`; exports return `caption_export_busy`; reviewed-row imports
return `caption_review_import_busy`; text-label and caption mutations return
`caption_mutation_busy`; metadata writes return `caption_metadata_busy`.
These guards consult both the current in-memory caption job registry and live
persisted set-and-forget runner locks, so a backend restart does not open a
short window where a still-running caption worker can be treated as idle.

Read-side consistency is important too. If image navigation or a scheduled
caption load tries to read the archive while a job is mutating it, the UI keeps
the current stable caption view and reports that the archive is busy. If the
read started before the job became active, the in-flight response is dropped
instead of repainting the caption textarea or alternate list from a stale
snapshot.

The reason is artifact coherence. Trainer JSONL, archive JSONL, review JSONL,
and report JSON are one export set. Exporting or importing while the underlying
caption records are changing can create a mixed packet that looks valid row by
row but is not coherent as a training dataset.

## Review Import Fail-Closed Behavior

Review import accepts JSONL rows, JSON arrays, wrapper objects with
`rows`/`review_rows`/`instruction_review_rows`, or a single review-row object.
It rejects scalar bodies and malformed wrappers.

It fails closed on:

- missing or mismatched dataset id;
- unsupported review decisions;
- unsupported actionable row origins;
- missing or mismatched QA ids;
- duplicate or conflicting review targets;
- stale caption0 or generated-QA text;
- stale selected training answers;
- missing image context;
- ambiguous matches;
- oversized or non-text review fields;
- review packets with no accepted, rejected, or needs-revision persisted
  language decisions.

Blank decisions are ignored. Deterministic metadata QA rows are skipped because
they are rebuilt from source labels during export rather than persisted as
language records.

## Artifact Readiness

The instruction report includes `training_readiness`:

- `ready`: structurally valid, artifact proofs agree, and selected rows are
  trainable under the configured review policy.
- `needs_review`: generated-language rows or quality gates require human
  attention before a default trainer export.
- `blocked`: selected rows are missing, malformed, inconsistent, rejected,
  needs-revision, or otherwise unsafe for training.

The UI refuses blocked trainer JSONL downloads. By default, it also refuses
needs-review trainer JSONL downloads through **Require ready report for trainer
JSONL**. API clients can request equivalent server-side enforcement with
`require_ready_instruction_export=true`.

Archive, review, and report downloads remain available for diagnostics even
when trainer JSONL is blocked.

## How A Reviewer Should Evaluate A Generated Packet

1. Open the instruction report first. Check readiness, blocking reasons,
   quality warnings, selected row count, review-row count, source-field
   provenance, export validation, and artifact consistency.
2. Open the archive JSONL. For a few images, verify that source annotations,
   generated language, deterministic QA, selected rows, and rejected rows are
   visibly separated.
3. Open the review JSONL. Confirm that generated-language rows have stable QA
   ids, candidate answers, selected training answers, validation status, review
   status, and blank review-decision fields.
4. Open the trainer JSONL. Confirm that it contains only model-visible
   `image_path`, `question`, `answer`, and optional audit metadata.
5. Import the trainer JSONL through the trainer loader. The loader should accept
   exactly the rows the report declares trainable.
6. Review generated-language rows manually. Import decisions. Re-export trainer
   JSONL with the ready gate enabled.
7. Run a small trainer-loader or fine-tuning dry run before treating the corpus
   as training-ready.

## Remaining Work Before Training Use

The implementation path is present and hardened, but a generated corpus still
needs a real-data pilot.

Required pilot steps:

1. Select a representative dataset slice.
2. Run **Create VLM training dataset** with set-and-forget enabled.
3. Package images, labels, trainer JSONL, archive JSONL, review JSONL, report
   JSON, and progress/run trace summary from the same run.
4. Manually review generated-language rows.
5. Import review decisions.
6. Re-export trainer JSONL with ready-report enforcement.
7. Run trainer-loader validation.
8. Run a small fine-tuning or loader-plus-batch dry run.
9. Inspect degraded recovery rate, rejected-row distribution, manual-review
   burden, and generated-QA usefulness.

Open questions for the pilot:

- What generated-QA mix gives the best training signal for the target corpus?
- How many generated QA rows per image are useful before review cost outweighs
  benefit?
- Should deterministic metadata QA be included in the final training mix?
- What manual-review threshold is acceptable before trainer export?
- What loop-recovery or deterministic-recovery rate is acceptable at scale?
- Are representative box subsets sufficient for dense scenes, or do specific
  classes need custom prompt coverage rules?

## Final Readiness Claim

The work completed here implements the requested application path: an operator
can launch dataset-scale caption-derived VLM training-data creation from the UI,
generate `caption0` plus configurable sub-captions as visual QA pairs, preserve
audit artifacts, review generated-language rows, import review decisions, and
export trainer JSONL behind browser/backend/trainer gates.

The implementation is deliberately conservative. It is designed to keep bad or
unreviewed rows out of trainer JSONL while preserving them for audit. The next
proof point is not more paperwork; it is a small real-data pilot that produces
the full artifact set, passes the trainer loader, and gives reviewers enough
evidence to decide whether the generated QA quality is acceptable for actual
fine-tuning.
