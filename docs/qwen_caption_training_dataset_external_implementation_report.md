# Qwen Caption Training Dataset External Implementation Report

Date: 2026-06-29

## Purpose

This report explains what was built in the caption-to-training-dataset work,
why it was built this way, what artifacts it produces, which guardrails now
exist, and what still must be proven before generated rows are used for
fine-tuning.

It is written for an external technical review team. The implementation is
dataset-neutral and intentionally avoids project-specific dataset names,
class-list assumptions, and private evaluation shortcuts.

## Executive Summary

The captioning stack now has two separate product paths:

- **Captioning for annotation assistance**: generate and manage image captions.
- **Instruction-dataset creation for VLM training**: generate auditable
  image/question/answer rows from a dataset, preserve provenance, support human
  review, and export trainer-ready JSONL only after structural gates pass.

The training-dataset path is exposed in the caption UI as
**Create VLM training dataset**. It can create a broad image-caption row
(`caption0`), configurable VLM-generated visual question/answer rows, and
optional deterministic metadata question/answer rows derived only from trusted
source labels.

The key design choice is strict evidence separation:

- Source labels remain trusted structured evidence.
- Captions and generated question/answer rows remain generated language.
- Deterministic metadata QA is generated by code from trusted labels.
- Flat trainer rows are selected exports, not the source of truth.

The result is an auditable production path for VLM training artifacts. It is not
a claim that every generated corpus is automatically training-grade. A reviewed
real-data pilot and trainer dry run are still required before using exported
rows for model training.

## What Was Built

### 1. A Real Training-Dataset Workflow In The UI

The caption panel now contains an explicit instruction-dataset section instead
of hiding training-data creation behind ordinary caption export controls.

The operator can configure:

- generated QA rows per image;
- generated QA mix;
- generated answer format;
- whether `caption0` rows are included;
- whether generated QA rows are included;
- whether deterministic metadata QA rows are included;
- whether read-only label context is passed to the generator;
- whether strict QA grounding is required;
- whether trainer JSONL download requires a ready instruction report.

The UI launch path refuses a job when every trainable row family is disabled.
If generated QA is requested but excluded from training output, the confirmation
text describes those candidates as archive/review-only.

### 2. Dataset-Scale Set-And-Forget Execution

Dataset captioning and instruction generation are routed through backend jobs
that persist progress, status, and artifacts. This is the default durable path
for dataset-scale work.

This matters because a training dataset can take long enough that a browser tab
or one direct model call is not a safe execution boundary. The current path
supports:

- backend job creation;
- progress polling;
- attach/recover behavior;
- persisted status and failure reports;
- active-job guards around artifact exports and review import;
- recovery-oriented messages when a job fails preflight, supervision, or pilot
  certification.

Direct single-image captioning remains available, but it is treated as a
diagnostic path rather than the recommended unattended dataset path.

### 3. Caption0 Plus Generated Visual QA

`caption0` is the broad per-image caption row. It provides a general
description prompt/answer pair for the image.

Generated visual QA rows are additional VLM-created language candidates about
the same image. They are useful because a single caption is usually too thin
for VLM fine-tuning. Multiple prompts can teach scene description, spatial
relationships, object presence, counts, visual attributes, and short grounded
answers.

Generated QA rows are not trusted as source labels. They are:

- parsed from the model output;
- structurally validated;
- deduplicated by image and question;
- stored with stable QA identifiers and provenance metadata;
- archived even when rejected from trainer output;
- subject to review metadata before final use.

### 4. Optional Deterministic Metadata QA

The workflow can also create deterministic QA rows from trusted labels. This is
optional and off by default because it changes the corpus mix.

Deterministic metadata QA is generated by code, not by the VLM. Supported
categories include:

- visible class list;
- object-count schema;
- per-class counts;
- positive class presence;
- negative class presence;
- simple spatial facts when source geometry supports them.

These rows are useful for training structured label-grounded answers, but they
must stay visibly separate from generated visual-language rows.

### 5. Four Exported Artifact Families

The workflow exports four artifact types.

`instruction_training_rows` is the trainer-facing JSONL. Each row is flat and
model-visible:

```json
{
  "image_path": "images/example.jpg",
  "question": "Describe this image in detail.",
  "answer": "A grounded answer.",
  "metadata": {
    "qa_id": "stable-row-id",
    "row_type": "generated_qa",
    "answer_source": "generated_qa_record",
    "answer_format": "natural",
    "validation_status": "accepted",
    "review_status": "accepted",
    "source_archive": "tator_caption_instruction_archive_v1"
  }
}
```

`instruction_archive_rows` are per-image audit records. They preserve source
annotation summaries, generated language records, deterministic QA, selected
training rows, rejected rows, validation status, review state, and consistency
proofs.

`instruction_review_rows` are candidate-level rows intended for human review.
They include row origin, row type, stable QA id, image path, question, candidate
answer, selected training answer, source summary, validation status, review
status, rejection reasons, and blank review decision/note fields.

`instruction_report` is the run-level report. It summarizes readiness,
quality metrics, row counts, rejection reasons, source-field provenance, artifact
consistency, and required actions.

### 6. Human Review Import

The workflow supports a closed review loop:

1. Export review JSONL.
2. Review generated-language rows outside the app.
3. Mark decisions such as accepted, rejected, or needs revision.
4. Import reviewed JSONL.
5. Re-export trainer JSONL.

Review import applies review metadata only. It does not edit source labels,
boxes, questions, answers, generated text, or final annotations.

The import path fails closed when rows are stale, ambiguous, duplicated,
targeting the wrong dataset, missing required context, or trying to create new
caption/generated-QA content through the review file.

### 7. Strict Export And Trainer Import Boundaries

The browser, backend, and trainer each validate the exported data instead of
trusting that another layer already did the right thing.

The browser blocks malformed or inconsistent downloads before writing files.
The backend can require `require_ready_instruction_export=true` so API clients
get the same ready-report gate as the UI. The trainer loader rejects stale,
hand-edited, malformed, review-pending, rejected, duplicate, or non-trainable
rows before fine-tuning starts.

This layered validation is intentional. Training files are easy to copy, edit,
or mix with stale artifacts. The final loader must be defensive even when the UI
and backend already validated an export.

## Why These Choices Were Made

### A Single Caption Is Not Enough For VLM Fine-Tuning

One broad caption can help, but it does not exercise the range of visual
questions a model may need to answer. A useful VLM dataset normally benefits
from varied prompts over the same image: broad descriptions, count questions,
presence questions, spatial questions, object attributes, and concise answers.

That is why the new path supports `caption0` plus configurable generated QA and
optional deterministic metadata QA.

### Generated Language Must Not Become Source Truth

Generated captions and generated QA can be wrong. Source labels are the trusted
structured evidence. The implementation therefore keeps generated language and
source annotations separate all the way through persistence, archive export,
review import, and trainer JSONL export.

When a generated answer makes a structured claim that can be checked against
source annotations, the final flattened answer can be rewritten from trusted
source data. When the claim cannot be checked, the row stays in the archive but
is rejected from trainer output.

### Flat Trainer Rows Need Rich Audit Files Beside Them

The trainer wants a simple `image_path`, `question`, `answer` row shape. That is
the right shape for loading and fine-tuning, but it is too thin to explain why a
row exists.

The archive and report preserve the missing context: source evidence, generated
candidates, validation decisions, selected rows, rejected rows, review state,
corpus metrics, and consistency proofs.

### Set-And-Forget Needs Real Guardrails

Long VLM dataset runs can hit model loops, oversized prompts, missing model
files, GPU/runtime failures, browser disconnects, or stale export attempts. The
workflow was hardened around those observed failure modes instead of assuming a
single happy path.

The current design makes failures visible and recoverable where possible:

- active dataset jobs block mutation/export actions that would race the archive;
- model availability is visible in the model dropdown;
- output-token budgets are computed from model capability, prompt size, and
  explicit user caps;
- dense label prompts use representative box subsets while authoritative counts
  remain complete;
- stream output is inspected for repeated-token loops;
- loop recovery can retry safely, unload the model runtime, or fall back through
  configured recovery policy;
- degraded recovery is recorded rather than silently treated as normal output.

### User Overrides Must Still Mean Overrides

Thinking-capable models can emit far more tokens than ordinary caption models.
Defaults therefore need to be high enough for those models, but a user-supplied
maximum output token value must remain a real cap.

The cleaned-up behavior separates:

- the UI/user-requested cap;
- the automatically selected effective budget;
- model-family defaults;
- prompt-size adaptation;
- per-stage recovery caps.

This is why logs may show an effective `max_new_tokens` value that differs from
an older visible UI default. The goal is to make that value explainable and to
ensure an explicit user override is not ignored.

## Runtime Hardening Summary

The captioning path was hardened after failures in dense and long-running
caption jobs.

### Prompt Size And Dense Box Handling

Authoritative counts remain complete. Box lists are now treated as prompt
context, not the source of count truth.

When a scene has many labels, the prompt builder can include a representative
spatial subset of boxes rather than dumping every box into the model prompt.
The wording makes clear that omitted boxes are not absent objects. This reduces
prompt size and lowers the chance of model loops or GPU stress while preserving
complete object counts.

### Dynamic Budgeting

The prompt builder records prompt size and effective token budgeting. Automatic
budgets can adapt when a rendered prompt becomes large. Explicit user caps
remain caps.

This solves the earlier confusion where one surface showed a user value while
the runtime used a larger stage default. The intended state is that logs expose
both the requested settings and the effective model call settings.

### Loop Detection

The system detects repeated output loops, including repeated punctuation or
token streams, trims the bad output, records loop events, and retries according
to recovery policy. The loop inspector reduces dependence on a single
`max_new_tokens` cutoff.

Token limits still remain necessary. They bound worst-case runtime and protect
against model/runtime behavior that cannot be recovered from within a single
call.

### Model Runtime Reloads

The full-image composition stage can appear slower because it carries more
context than a crop stage: full image, source counts, representative spatial
context, window observations, and merge instructions. That does not necessarily
mean the model was reloaded.

Reloads happen when the requested model changes, when a recovery path unloads
after a loop, or when the runtime must recover from a known bad state. A normal
full-image call can still take longer simply because the prompt and image are
larger.

### Process-Level GPU Failures

Some Metal/GPU failures abort the Python process and cannot be caught as normal
Python exceptions. The practical strategy is:

- keep jobs persisted;
- isolate long-running model work behind backend supervision;
- make restart/resume behavior explicit;
- record enough state to avoid accepting partial bad output as valid output.

This does not make process-level GPU faults impossible. It makes the product
path less dependent on a single unobservable in-memory call.

## UI/UX Hardening Summary

The UI was updated so an operator can understand and operate the new workflow
without needing to inspect raw logs.

Important UI behavior now includes:

- a distinct **Create VLM training dataset** action;
- set-and-forget controls and explanatory status;
- direct-caption diagnostics separated from dataset jobs;
- model dropdown coloring where missing/download-needed models are visually
  distinct from local models;
- wrapped button rows so caption/instruction actions do not clip in narrow
  panels;
- busy-state locks around caption exports, review import, glossary edits,
  caption mutation, dataset download, and dataset deletion;
- operator-readable messages for active job locks and review-import failures;
- structured failure summaries for pilot certification, backend supervision,
  and preflight failures;
- readiness status before trainer JSONL download.

The goal is not only correctness. The goal is that a user can run the dataset
path without guessing whether a disabled button, failed import, missing model,
or blocked export is expected.

## What Is Ready Now

The implementation is ready for external implementation review and a small
real-data pilot.

Ready means:

- the UI can launch the instruction-dataset path;
- the backend can create the instruction artifacts;
- the artifacts preserve provenance;
- the review loop exists;
- strict export gates exist;
- the trainer can import the flat rows;
- malformed or stale artifacts fail closed at multiple layers.

Ready does not mean every generated row is semantically useful for training.
That must be measured on real data.

## What Still Must Be Proven

Before using exported rows for fine-tuning, run a pilot that includes:

- a dense labeled scene;
- an image with no source objects;
- an image with missing source labels;
- multiple object classes;
- existing alternate captions;
- small objects near crop or image boundaries;
- at least one long set-and-forget run;
- at least one intentionally rejected and one needs-revision review decision;
- a strict re-export after review import;
- a trainer import or small fine-tuning dry run.

The pilot should answer:

- Are generated QA questions useful, non-redundant, and grounded?
- Does strict grounding reject the right rows?
- Does review import preserve source labels and generated text?
- Does the final trainer JSONL match the archive and report?
- Does the trainer loader reject deliberately malformed variants?
- Is the rate of loop recovery or degraded captions acceptable?
- Are the UI failure messages actionable for non-developer operators?

## Reviewer Acceptance Criteria

Treat the implementation as acceptable for a pilot if the reviewer can confirm:

- **Workflow**: one UI action can launch dataset-scale instruction generation
  after settings are chosen.
- **Evidence separation**: generated language never overwrites source labels.
- **Artifact completeness**: trainer JSONL, archive JSONL, review JSONL, and
  report JSON all export and agree.
- **Review loop**: review decisions import as metadata only and affect
  subsequent row selection.
- **Fail-closed behavior**: stale, malformed, rejected, review-pending,
  duplicate, or inconsistent rows do not reach trainer import.
- **Runtime visibility**: model availability, active-job locks, loop recovery,
  and set-and-forget status are visible to the operator.
- **Training boundary**: the trainer imports valid final rows and rejects
  deliberately invalid variants.

Treat the implementation as not accepted for training use if a generated corpus
has not completed the pilot and review loop.

## Design Risks And Mitigations

| Risk | Mitigation |
| --- | --- |
| Generated QA contains hallucinated structured facts | Rewrite supported structured answers from source labels or reject unsupported rows from trainer output |
| Flat trainer JSONL loses provenance | Preserve archive JSONL, review JSONL, report JSON, and row metadata |
| Review file edits create stale or ambiguous targets | Match by dataset identity, stable QA id, image context, and current reviewed text |
| Dense label prompts overload the model | Use representative spatial box subsets while keeping full counts authoritative |
| Thinking-capable models need more output budget | Use high automatic defaults with prompt-size adaptation while preserving explicit user caps |
| Model output loops look like hangs | Inspect live output, trim loops, retry safely, unload bad runtime state, and record recovery |
| Backend job mutates artifacts during export | Block export/import/mutation actions while active caption dataset jobs are running |
| Process-level GPU failures abort Python | Persist job state and use supervised restart/resume rather than relying on catchable exceptions |
| UI hides missing model state | Color missing/download-needed model choices distinctly from local models |

## Bottom Line

The work converts caption generation from a single-output annotation assistant
into an auditable VLM training-data production path. The important part is not
only that JSONL files are produced. The important part is that each trainable
row can be traced to image identity, source evidence, generated language,
validation state, review state, artifact consistency, and final trainer import
checks.

The next milestone is not another structural export feature. The next milestone
is a reviewed real-data pilot that proves generated QA quality, review
throughput, recovery-rate acceptability, and trainer-loader compatibility on the
actual target corpus.
