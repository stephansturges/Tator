# Caption-Derived VLM Training Dataset Partner Technical Handoff

Date: 2026-06-29

## Purpose

This document is the standalone technical handoff for the caption-derived VLM
training-dataset work. It is written for an external technical reviewer who
needs to understand what was built, why it was built this way, how an operator
uses it, which artifacts are produced, which validation gates protect the
output, which failure modes were hardened, and what proof is still required
before a generated corpus should be used for fine-tuning.

The document is intentionally dataset-neutral. Do not add customer, partner,
dataset, or private project code names to this file. The implementation should
be evaluated from its product behavior, data contracts, provenance model,
validation gates, and test evidence.

## Executive Summary

The captioning stack now has two separate product workflows:

- **Caption assistance**: generate, store, compare, promote, and export
  captions for annotation work.
- **VLM training-dataset creation**: generate auditable
  image/question/answer rows from a selected dataset, preserve source evidence,
  support human review, and export trainer-ready JSONL only after validation
  gates pass.

The training workflow is launched from the UI with **Create VLM training
dataset**. It can produce:

- one broad per-image `caption0` row;
- configurable VLM-generated visual question/answer rows per image;
- optional deterministic metadata QA rows generated from trusted labels;
- flat trainer JSONL;
- per-image instruction archive JSONL;
- candidate-level review JSONL;
- run-level instruction report JSON;
- a self-contained training bundle ZIP with copied images, effective labels,
  JSONL/JSON artifacts, `labelmap.txt`, and a checksum manifest.

The central design decision is evidence separation. Source labels remain
trusted structured data. Captions and generated QA remain generated language.
Deterministic metadata QA is generated by code from trusted labels. Trainer
rows are selected exports from those layers, not a replacement source of truth.

The current implementation is ready for external implementation review and a
small real-data pilot. It is not a claim that every generated corpus is already
fine-tuning-grade. A corpus should be used for training only after a pilot run,
human review where required, reviewed-row import, strict re-export,
trainer-loader validation, and at least a loader-plus-batch or fine-tuning dry
run.

## End-To-End Operator Path

The expected operator path is:

1. Open the dataset in the labeling UI.
2. Select the caption model and confirm it is locally available.
3. Configure captioning style, prompt limits, generated QA settings,
   deterministic metadata QA settings, review requirements, and strict export
   behavior.
4. Click **Create VLM training dataset**.
5. Let the backend job run with set-and-forget recovery enabled where
   configured.
6. Reattach to the running or completed job after browser refresh if needed.
7. Inspect the readiness summary, report metrics, rejected rows, recovery
   events, and generated row counts.
8. Download the self-contained training bundle.
9. Review generated-language candidates using the review JSONL.
10. Import review decisions.
11. Re-export trainer JSONL or a new bundle with strict readiness enabled.
12. Run trainer-loader validation and a loader-plus-batch or fine-tuning dry
    run before treating the corpus as ready.

There are intentionally two export moments. The first bundle gives reviewers
the raw candidate and evidence package. The second strict export is the
candidate training input after review decisions have been applied.

## What Was Built

### Dedicated UI Workflow

The UI exposes a distinct **Create VLM training dataset** action near the
caption batch controls. It is not hidden behind ordinary caption export.

The operator can configure:

- generated QA pairs per image;
- generated QA mix;
- generated answer format;
- whether `caption0` is included in trainer JSONL;
- whether generated QA is included in trainer JSONL;
- whether deterministic metadata QA is included;
- whether read-only label context is passed to generation;
- strict QA grounding;
- whether trainer JSONL export requires a ready instruction report.

The UI refuses to launch a training-dataset job when every trainable row family
is disabled. It also distinguishes archive/review-only generated QA from rows
that will actually be flattened into trainer JSONL.

### Caption0 Plus Multi-Prompt QA

`caption0` is the broad scene-description row for an image. Generated visual QA
rows are additional VLM-created question/answer candidates for the same image.

This exists because one caption per image is usually too thin for VLM
fine-tuning. The same image can now contribute rows for broad description,
object presence, counts, spatial relationships, attributes, and shorter
grounded answers.

Generated QA rows are candidates. They are parsed, structurally validated,
deduplicated, archived, exposed for review, and then selected or rejected. They
do not overwrite labels, boxes, dataset state, or source annotations.

### Optional Deterministic Metadata QA

The workflow can also generate deterministic metadata QA from trusted labels.
This option is explicit and configurable because it changes the corpus mix.

Examples include:

- visible class list;
- object-count schema;
- per-class counts;
- positive class presence;
- negative class presence;
- simple spatial facts from source geometry when supported.

These rows are not VLM-generated. They are code-generated from labels and kept
as their own row origin.

### Self-Contained Training Bundle

The preferred external handoff artifact is **Download training bundle**. The
bundle contains:

- copied image bytes under `images/...`;
- effective label files under `labels/...`;
- `caption_instruction_training.jsonl`;
- `caption_instruction_archive.jsonl`;
- `caption_instruction_review.jsonl`;
- `caption_instruction_report.json`;
- `labelmap.txt`;
- `caption_instruction_bundle_manifest.json`.

The bundle rewrites exported row `image_path` values to the copied `images/...`
paths and preserves original image paths and image SHA-256 values in metadata.
This makes the handoff self-contained while keeping provenance back to the
original local dataset.

The backend validates the bundle before returning it. The manifest must declare
the expected format, manifest path, checksum scope, file count, non-manifest
ZIP member inventory, required artifact paths, file roles, row counts,
image/label asset inventories, byte counts, SHA-256 digests, and recomputed
trainer/archive/review/report consistency. Duplicate ZIP members, missing
members, malformed JSONL artifacts, mismatched row counts, wrong artifact
roles, stale image/label inventory entries, mismatched artifact evidence,
mismatched byte counts, and checksum mismatches fail the download. Trainer,
archive, and review rows must also resolve to copied images inside the bundle;
stale rows that still point at an uncopied original path block the bundle
instead of producing a partially self-contained handoff.

### Coordinated JSONL And Report Artifacts

Every complete instruction-dataset run can produce four coordinated artifact
families:

| Artifact | Purpose |
| --- | --- |
| Trainer JSONL | Flat `image_path` / `question` / `answer` rows consumed by training |
| Instruction archive JSONL | Per-image construction records with source annotations, generated candidates, selected rows, rejected rows, and consistency metadata |
| Review JSONL | Candidate-level rows for human review and metadata-only review import |
| Instruction report JSON | Run-level metrics, validation state, readiness state, settings fingerprint, and blocking reasons |

Trainer JSONL is the model-visible file, but it is not sufficient audit
evidence by itself. The archive, review file, report, and bundle manifest
explain why rows exist, what they were derived from, whether they were reviewed,
and whether the referenced pixels are the expected bytes.

### Closed Human Review Loop

The operator can:

1. Export review JSONL.
2. Review generated-language candidates outside the app.
3. Import accepted, rejected, or needs-revision decisions.
4. Re-export trainer JSONL with readiness gates enabled.

Review import is metadata-only. It applies review decisions and reviewer notes
to matching `caption0` and generated-QA records. It does not edit labels,
boxes, image paths, generated questions, generated answers, deterministic QA,
or final annotations.

### Trainer Loader Validation

The trainer imports flat rows but retains enough metadata to reject unsafe
input. Rows are rejected when they are malformed, missing required provenance,
marked rejected or needs-revision, invalid for their expected answer format,
duplicated after canonical normalization, or unable to resolve their image
path.

This keeps trainer input simple while making the handoff fail closed.

## Implementation Map

The feature is split across browser UI, backend APIs, persisted archive state,
export validators, bundle creation, and tests. The important implementation
boundaries are:

| Area | Responsibility |
| --- | --- |
| Labeling UI | Shows model availability, instruction settings, launch controls, readiness summaries, review import/export, and bundle downloads |
| Caption job launcher | Normalizes settings, blocks unsafe row-family combinations, starts backend jobs, and preserves active-job state |
| Backend caption job | Runs image captioning, generated QA creation, deterministic metadata QA generation, retries, recovery, and progress recording |
| Caption archive | Stores per-image caption and instruction records without mutating source annotations |
| Instruction archive builder | Produces per-image source/language/provenance records from caption records and source labels |
| Trainer exporter | Flattens selected rows into simple image/question/answer JSONL |
| Review exporter/importer | Exposes generated-language candidates and imports metadata-only review decisions |
| Report builder | Records metrics, settings fingerprint, validation, readiness, blockers, and row-family counts |
| Bundle exporter | Copies images and labels, rewrites paths, writes all artifacts, and validates manifest integrity |
| Trainer loader | Rejects malformed, duplicate, unresolved, rejected, or unsupported rows before training |
| Test suite | Covers row construction, validation, review, bundle packaging, UI contract, and failure formatting |

This split is deliberate. The UI can explain and block unsafe operations, but
the backend and trainer loader still validate independently because artifacts
can be moved, edited, or generated through API/script paths.

## What Changed During Hardening

The hardening work focused on specific failures observed during local use and
on expected dataset-scale failure modes:

- **Output token confusion**: prompt preview, logs, and runtime now distinguish
  user-requested caps from effective runtime budgets. Auto mode can choose
  larger defaults for thinking-capable models, while explicit numeric user
  overrides remain hard caps.
- **Excessive prompt size**: authoritative counts are retained, but dense box
  context is represented rather than blindly dumping every box into every
  full-image prompt. Prompt budget telemetry makes the effective prompt size
  visible.
- **Repeated-token stalls**: streaming loop detection watches live output,
  trims repeated surfaces for display, raises controlled loop failures, and can
  retry with safer decode settings.
- **Fragile full-image composition**: detailed/windowed runs can compose from
  completed crop evidence instead of forcing one more full-image visual call in
  the most fragile stage.
- **Runtime state after loops**: repeated-loop recovery can unload the model
  before a safe retry so a bad generation state is not silently reused.
- **GPU or worker aborts**: long dataset jobs are tracked as backend jobs with
  persisted state and process-level recovery surfaces, so an aborted ML worker
  becomes an observable failed or recoverable job state instead of a silent UI
  hang.
- **Manual recovery burden**: set-and-forget behavior was made explicit. The
  user can still override settings, but large runs are designed to resume,
  recover, and report without requiring manual intervention after every bad
  model call.
- **Model availability ambiguity**: model selectors distinguish locally
  available models from models that need downloading.
- **Archive races**: caption generation, instruction generation, review import,
  export, bundle download, edit/autosave, and archive deletion now share busy
  guards so active jobs cannot be mixed with stale reads or competing writes.
- **Incomplete bundles**: bundle export now fails closed when trainer, archive,
  or review rows cannot be rewritten to copied images inside the ZIP.
- **Weak report parity**: backend artifact validation now checks readiness
  schema and state, matching the browser preflight expectation instead of
  trusting a malformed or contradictory report.
- **Raw error messages**: UI failure formatting turns backend export and bundle
  errors into operator-facing readiness, image, manifest, corruption, and
  consistency messages.

## Why The Design Looks This Way

### Product Goal

The goal is not "generate captions." The goal is to create a traceable training
dataset from images, source annotations, generated language, and review
decisions without corrupting annotation state. The generated corpus must be
useful to a VLM trainer, but also auditable by a data owner.

### Generated Text Must Not Become Source Truth

Generated captions and answers can be plausible and wrong. The workflow
therefore keeps label-derived evidence in `source_annotations` and generated
language in `language_annotations`.

Allowed flow:

```text
trusted labels and deterministic geometry
  -> source_annotations
  -> deterministic metadata QA
  -> selected flat trainer rows

image plus read-only context
  -> generated language candidates
  -> validation and review
  -> selected flat trainer rows
```

Forbidden flow:

```text
generated caption
  -> source_annotations.object_counts

generated answer
  -> source_annotations.visible_classes

review metadata
  -> hidden training signal
```

If a fact should train the model, it must appear in the trainer row's
`question` or `answer`, not only in metadata.

### Source And Language Provenance

The archive keeps source-derived and language-derived data in different
families:

| Family | Source | Training use |
| --- | --- | --- |
| Source annotations | Existing labels and geometry | Audit evidence and deterministic QA input |
| `caption0` | VLM broad caption | Optional broad trainer row |
| Generated QA | VLM-created question/answer candidates | Optional trainer rows after validation and review policy |
| Deterministic metadata QA | Code-generated from trusted labels | Optional trainer rows when source evidence supports them |
| Review metadata | Human decisions and notes | Selection/rejection gate, not hidden model signal |

This means a trainer row can be traced back to one of three row origins:
`caption0`, `generated_qa`, or `deterministic_metadata_qa`. The origin matters
because each row family has different trust and review requirements.

### What This Does Not Do

The implementation intentionally does not:

- mutate boxes, labels, or final annotations during dataset generation;
- infer source object counts from generated captions;
- train on hidden metadata fields;
- treat an empty label file as a reason to invent label-derived facts;
- certify a full fine-tuning corpus without a reviewed pilot and trainer dry
  run;
- guarantee that every generated visual QA candidate is true without review;
- bypass backend or trainer-loader validation just because the browser preflight
  passed;
- make direct one-off captioning the default path for unattended dataset-scale
  work.

Those constraints protect the data owner and keep the output usable for
training without silently changing the meaning of the source dataset.

### Flat Trainer Rows Need Rich Evidence Beside Them

The trainer should load a simple shape:

```json
{"image_path":"images/example.jpg","question":"Describe this image in detail.","answer":"A grounded answer."}
```

That shape is correct for training, but too thin for audit. The archive,
review file, report, and manifest preserve source labels, generated
candidates, row origins, validation results, review decisions, selected rows,
rejected rows, readiness blockers, copied image paths, original image paths,
and checksums.

### Empty Labels Are A Valid State, Not A Prompt To Hallucinate

An empty label file can be valid when an image has no annotated objects. It can
also be an unexpected dataset issue. The implementation treats that distinction
as data provenance, not as a reason to ask the VLM to invent label-specific
facts.

When source evidence is missing or empty, the workflow records the source state
and avoids promoting unsupported label-specific questions into trainer rows.
The VLM may still describe the visible image, but deterministic count, class,
or label-derived rows require supporting source evidence. A real-data pilot
should still audit empty labels to decide whether they are intentional negative
examples or dataset preparation mistakes.

### Long Jobs Need Set-And-Forget Behavior

Dataset-scale generation cannot depend on a single browser session or one
perfect model call. During hardening, the observed and anticipated failure
modes included oversized prompts, confusing output-token traces, repeated-token
loops, full-image composition stalls, missing model files, backend restarts,
and MLX/Metal GPU failures.

The implementation therefore uses backend jobs, persisted job state, progress
polling, attach/recover behavior, active-job guards, prompt-budget telemetry,
streaming loop inspection, bounded retry policy, runtime unload on repeated
output loops, and process-level isolation for dataset jobs.

### Recovery Must Be Automatic But Observable

Set-and-forget mode is not intended to make an operator manually recover every
failure. It should run unattended, recover when configured recovery paths are
allowed, and preserve enough telemetry for the operator to see when output was
degraded or retried.

Recovery events are recorded in API responses, saved caption records, job
logs, progress output, and reports. A final caption or training row should not
look identical whether it came from a clean visual model call, a safe retry, a
fallback model, text-only window-evidence composition, or deterministic
count/layout recovery.

## Set-And-Forget Behavior

Set-and-forget mode is the correct default for dataset-scale generation. It is
not the same thing as ignoring failures. It means:

- active jobs are backend-owned and visible after refresh;
- progress and health are polled from the backend;
- failed model calls can be retried under bounded policies;
- loop recovery and safe decode retries are automatic when enabled;
- large launches can require pilot certification and backend crash supervision;
- completion is accompanied by explicit health, readiness, and validation
  summaries;
- blocked jobs preserve the reason rather than pretending to have succeeded.

For small diagnostics, direct single-image captioning remains available, but it
is treated as deliberate interactive use rather than the primary training data
path.

## Runtime And Prompt Hardening

| Risk | Hardened behavior |
| --- | --- |
| User output cap did not match runtime cap | UI separates Auto from explicit numeric override; logs record requested and effective budgets |
| Thinking-capable models need larger defaults | Auto budgets can be higher for thinking-capable models while user numeric caps remain hard caps |
| Dense scenes made prompts too large | Counts remain authoritative; dense box context is represented by a capped spatial subset |
| Large prompts can crowd output space | Prompt preview and runtime prompt measurement adapt non-explicit output budgets |
| Repeated punctuation looked like a hang | Streaming loop inspection detects repeated surfaces, trims progress display, raises a controlled loop error, and routes recovery |
| Loop recovery can leave runtime state suspect | Recovery can unload the model before retrying with safer decode settings |
| Full-image visual composition could stall | Windowed set-and-forget mode can compose from completed crop evidence instead of resending the full image tensor |
| GPU faults can abort the Python worker | Dataset runs use isolated child attempts and persisted status so process aborts become observable job failures |
| Browser refresh loses progress | Job metadata, progress, attach/recover, and default-root discovery mirrors preserve job visibility |
| Model availability was ambiguous | Model selectors visually distinguish download-needed models from locally available models |
| Export errors were too raw | Bundle/export failures are formatted into operator-facing readiness, packaging, manifest, image, and corruption messages |

The full-image stage does not intentionally reload the model just because it is
the full image. It can feel slower because it has the full image tensor and the
largest prompt context. Model unload/reload is expected only for explicit
runtime unload actions, model switches, or recovery paths such as repeated-loop
cleanup. In set-and-forget windowed mode, the safer default avoids the most
fragile full-image visual composition by composing from completed window
evidence where possible.

## Archive And Concurrency Hardening

Caption and instruction jobs mutate the caption archive. The UI and backend
therefore treat the archive as a shared resource with explicit busy-state
guards.

Implemented protections include:

- caption image, caption next-N, caption all, and training-dataset launch paths
  check archive-idle state at action time;
- caption output editing and autosave are blocked while a live job mutates the
  archive;
- prompt, model, token, decode, glossary, style, recipe, row-family, and
  instruction settings lock during active jobs;
- current-image caption archive reads defer while the archive is mutating;
- export, bundle download, review import, archive deletion, and related
  dataset operations check active jobs before reading or mutating shared files;
- live-job busy responses are shown as blocked operator states rather than
  generic failures.

The purpose is to prevent stale exports, mixed settings, archive races, and
silent UI mutation while a long-running backend job owns the dataset archive.

## UI And UX Contract

An external reviewer should expect these operator-facing behaviors:

- model choices communicate whether the model is available locally;
- instruction settings are visible near the caption batch controls;
- launch is blocked when no trainable row family is enabled;
- generated QA can be archive/review-only when excluded from trainer JSONL;
- long-running dataset jobs show progress and can be reattached after refresh;
- caption, prompt, glossary, dataset, recipe, archive, export, and review-import
  controls lock while a caption or instruction job mutates the archive;
- stale click or input events do not silently mutate active run settings;
- reviewed-row import is blocked while the archive is busy and checked again at
  selected-file import time;
- long status text wraps without squeezing action buttons;
- bundle and export failures report actionable validation messages, not raw
  internal backend codes.

The intended operator experience is: choose settings, start the training
dataset job, walk away, come back, inspect status, download a bundle, review
rows, import decisions, and re-export trainer JSONL without babysitting each
model call.

## Review Import Contract

Review import is deliberately narrow. A reviewer may submit decisions and
notes, but may not use review JSONL as an editing backdoor for generated text,
image paths, source annotations, or deterministic rows.

Supported decisions are applied only when the row can be matched to the current
dataset archive and the submitted text still matches the current candidate.
The import path rejects:

- wrong dataset identity;
- malformed rows;
- unsupported row origins;
- duplicate actionable targets;
- stale `caption0` text;
- stale generated QA question or answer text;
- generated QA rows that no longer exist;
- generated QA rows with missing current text;
- deterministic metadata QA review attempts that would imply mutation;
- review files that are too large to read safely in the browser.

This is what makes the loop closed rather than just export-only. A reviewer can
mark candidates accepted, rejected, or needs-revision, and the next strict
export reflects that metadata without giving the review file authority over
source data.

## Artifact Contract

### Trainer Row

Trainer JSONL rows are flat and model-visible:

```json
{
  "image_path": "images/train/frame_0001.jpg",
  "question": "Describe this image in detail.",
  "answer": "A grounded visual answer.",
  "metadata": {
    "qa_id": "stable-row-id",
    "row_type": "generated_qa",
    "answer_source": "generated_qa_record",
    "validation_status": "accepted",
    "review_status": "accepted",
    "source_archive": "tator_caption_instruction_archive_v1",
    "original_image_path": "train/frame_0001.jpg",
    "image_sha256": "..."
  }
}
```

The trainer should use only `image_path`, `question`, and `answer` as model
signal. Metadata is for validation, audit, and loader rejection, not hidden
training signal.

### Archive Row

Archive JSONL rows are per-image construction records. They preserve:

- source annotations and source-evidence state;
- `caption0`;
- generated QA candidates;
- deterministic metadata QA;
- selected flattened training rows;
- rejected candidates and reasons;
- settings fingerprint;
- artifact consistency metadata.

### Review Row

Review JSONL rows are candidate-level records for human decisions. They include
stable ids, image paths, row origin, current text, selected training answer,
validation state, review state, rejection reasons, source summary, and
reviewer-decision fields.

Review import applies metadata decisions only. It rejects stale text,
duplicate actionable targets, wrong dataset identity, unsupported origins,
malformed rows, and content-mutating attempts.

### Report

The report records:

- row counts and row-type distribution;
- split-level metrics;
- rejection reasons;
- readiness status and blocking reasons;
- source-field provenance;
- validation summaries;
- prompt-budget and recovery telemetry where available;
- instruction settings and settings fingerprint;
- archive/review/trainer consistency.

### Bundle Manifest

The bundle manifest records enough information for an external receiver to
verify the handoff without relying on the original machine:

- manifest format;
- declared artifact paths;
- ZIP member list excluding the manifest itself;
- role for each artifact;
- byte count for each artifact;
- SHA-256 digest for each artifact;
- copied image asset inventory;
- copied label asset inventory;
- original image path and copied bundle path mapping;
- original image SHA-256 when available;
- checksum scope and manifest self-description.

The bundle is rejected if these claims do not match the actual ZIP contents.
This is intentionally stricter than simply creating a ZIP because the bundle is
the handoff boundary to the training team.

## Validation Boundaries

The workflow validates at multiple layers because training artifacts can be
copied, edited, mixed, or shared outside the UI.

| Layer | What it protects |
| --- | --- |
| Browser preflight | Row shape, metadata, readiness schema and count parity, embedded consistency count parity, review state, duplicates, settings fingerprint, and consistency before writing downloads |
| Backend export validation | Flat trainer rows, report readiness schema, state, and count parity, embedded consistency count parity, settings fingerprint, archive-row shape, review-row shape, image alias resolution, and strict API/script export |
| Bundle validation | Copied image/label assets, rewritten trainer/archive/review paths, trainer rows, artifact consistency, ZIP integrity, manifest inventory, required artifact paths, JSONL row-count parity, image/label inventory parity, file roles, byte counts, SHA-256 digests, and recomputed artifact consistency from the finished ZIP |
| Review import validation | Wrong dataset, stale text, duplicate targets, unsupported origins, malformed rows, and metadata-only mutation |
| Trainer loader validation | Missing images, malformed rows, duplicate canonical image/question pairs, rejected or needs-revision rows, invalid answer format, missing provenance |

This redundancy is intentional. The trainer loader remains the last defense
even when the UI and backend have already validated the export.

For an external review, provide:

- this handoff document;

- the instruction report;

## Required Pilot Before Training Use

The implementation is ready for review and pilot use, but a generated corpus
should not be treated as training-ready until the following proof exists:

1. Run the workflow on a representative real-data pilot subset.
2. Download the self-contained training bundle from that run.
3. Inspect the report for readiness state, rejected-row reasons, recovery
   rates, deterministic recovery cases, and artifact consistency.
4. Review generated-language rows in the review JSONL.
5. Import review decisions.
6. Re-export trainer JSONL with strict readiness enabled.
7. Load the final JSONL with the trainer loader.
8. Run at least a loader-plus-batch or fine-tuning dry run.
9. Record reviewer throughput and generated-QA usefulness.
10. Decide whether deterministic metadata QA should stay enabled for the final
    corpus mix.

The pilot should explicitly inspect:

- empty label files;
- missing-label source states;
- generated QA that makes structured count or class claims;
- rows rewritten from trusted source annotations;
- rejected and needs-revision rows;
- any deterministic recovery rows;
- any fallback-model rows;
- any bundle manifest mismatch or image-resolution failure.
