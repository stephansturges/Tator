# Caption-Derived VLM Training Dataset External Delivery Report

Date: 2026-06-29

## Purpose

This document is the complete external delivery report for the
caption-derived VLM training-dataset work. It explains what was implemented,
why the implementation is structured this way, what an operator can do in the
UI, which artifacts are produced, which validation gates protect the output,
which runtime and UX hardening was added, and what still must be proven before
using a generated corpus for fine-tuning.

The document is intentionally dataset-neutral. Do not add customer, partner,
dataset, or project code names before sharing it. The implementation can be
reviewed from its product behavior, data contracts, provenance model,
validation gates, and test evidence without private names.

## Executive Summary

The captioning stack now supports two separate product workflows:

- **Caption assistance**: generate, store, compare, promote, and export
  captions for annotation work.
- **Training-dataset creation**: create auditable image/question/answer rows
  from a selected dataset, preserve source evidence separately from generated
  language, support human review, and export trainer-ready JSONL only after
  validation gates pass.

The training workflow is launched from the UI with **Create VLM training
dataset**. It can produce:

- a broad `caption0` row per image;
- configurable VLM-generated question/answer candidates per image;
- optional deterministic metadata QA rows generated from trusted labels;
- flat trainer JSONL;
- per-image instruction archive JSONL;
- candidate-level review JSONL;
- a run-level instruction report JSON.

The central design choice is evidence separation. Source labels remain trusted
structured data. Captions and generated QA remain generated language.
Deterministic metadata QA is generated by code from trusted labels. Trainer
rows are selected exports from those layers, not a new source of truth.

The current implementation is ready for external technical review and a small
real-data pilot. It is not a claim that every generated corpus is already
fine-tuning-grade. A corpus should be used for training only after a pilot run,
review where required, reviewed-row import, strict re-export, trainer-loader
validation, and at least a loader-plus-batch or fine-tuning dry run.

## Original Product Gap

The old caption path could generate useful per-image prose, but a VLM
fine-tuning dataset needs more than one caption string:

- multiple question styles over the same image;
- source evidence that can be audited independently of generated text;
- stable row ids for review and import;
- row-level validation status;
- review decisions that do not mutate source labels;
- a flat trainer input file;
- a richer archive and report explaining where each training row came from;
- durable execution for long jobs that may outlive a browser tab.

The implemented workflow closes that gap without turning generated language
into trusted source annotations.

## What Was Implemented

### 1. Dedicated UI Workflow

The UI exposes a distinct **Create VLM training dataset** action near the
caption batch controls. It is not hidden inside ordinary caption export.

The operator can configure:

- generated QA pairs per image;
- generated QA mix;
- generated answer format;
- whether `caption0` is included in trainer JSONL;
- whether generated QA is included in trainer JSONL;
- whether deterministic metadata QA is included;
- whether read-only label context is passed to generation;
- strict QA grounding;
- whether trainer JSONL export requires a ready instruction report.

The UI refuses to launch a training-dataset job when every trainable row family
is disabled. It also distinguishes archive/review-only generated QA from rows
that will actually be flattened into trainer JSONL.

### 2. Caption0 Plus Multi-Prompt QA

`caption0` is the broad image-description row. Generated QA rows are additional
VLM-created question/answer candidates for the same image.

This exists because one caption per image is usually too thin for VLM
fine-tuning. The new workflow can create rows that cover broad scene
description, object presence, counts, spatial relationships, and concise
grounded answers.

Generated QA rows are candidates. They are parsed, deduplicated, validated,
archived, exposed for review, and selected or rejected. They do not overwrite
source labels or become source annotations.

### 3. Optional Deterministic Metadata QA

The workflow can also generate deterministic metadata QA from trusted labels.
This is explicit and configurable because it changes the corpus mix.

Examples include:

- visible class list;
- object-count schema;
- per-class counts;
- positive class presence;
- negative class presence;
- simple spatial facts from source geometry when supported.

These rows are not VLM-generated. They are code-generated from labels and are
kept as their own row origin.

### 4. Self-Contained Bundle Plus Coordinated Artifacts

Every complete instruction-dataset run can produce one self-contained training
bundle and four coordinated artifact families:

| Artifact | Purpose |
| --- | --- |
| Training bundle ZIP | Copied image bytes, effective labels, trainer JSONL, archive JSONL, review JSONL, report JSON, `labelmap.txt`, and checksum manifest |
| Trainer JSONL | Flat `image_path` / `question` / `answer` rows consumed by training |
| Instruction archive JSONL | Per-image construction records with provenance, candidates, selected rows, rejected rows, and consistency metadata |
| Review JSONL | Candidate-level rows for human review and metadata-only review import |
| Instruction report JSON | Run-level metrics, validation state, readiness state, and blocking reasons |

The training bundle is the preferred handoff because its trainer rows point to
copied `images/...` paths inside the zip and carry original-image path plus
SHA-256 metadata. Trainer JSONL is the model-visible file, but it is not enough
to audit the corpus. The archive, review file, report, and manifest explain why
rows exist, what they were derived from, and whether the referenced pixels are
the expected bytes. Bundle download also validates the manifest's file
inventory, byte counts, and SHA-256 digests before returning the ZIP.

### 5. Closed Human Review Loop

The operator can:

1. Export review JSONL.
2. Review generated-language candidates outside the app.
3. Import accepted, rejected, or needs-revision decisions.
4. Re-export trainer JSONL with readiness gates enabled.

Review import is metadata-only. It applies review decisions and reviewer notes
to matching `caption0` and generated-QA records. It does not edit labels,
boxes, image paths, generated questions, generated answers, deterministic QA,
or final annotations.

### 6. Trainer Loader Validation

The trainer imports flat rows but keeps enough metadata to reject unsafe input.
Rows are rejected when they are malformed, missing required provenance, marked
rejected or needs-revision, invalid for their expected answer format,
duplicated after canonical normalization, or unable to resolve their image
path.

This keeps trainer input simple while making the handoff fail closed.

## Why The Design Looks This Way

### Source Truth Must Stay Separate From Generated Text

Generated captions and answers can be plausible and wrong. The workflow
therefore keeps label-derived evidence in `source_annotations` and generated
language in `language_annotations`.

Allowed flow:

```text
trusted labels and deterministic geometry
  -> source_annotations
  -> deterministic metadata QA
  -> selected flat trainer rows

image plus read-only context
  -> generated language candidates
  -> validation and review
  -> selected flat trainer rows
```

Forbidden flow:

```text
generated caption
  -> source_annotations.object_counts

generated answer
  -> source_annotations.visible_classes

review metadata
  -> hidden training signal
```

If a fact should train the model, it must appear in the trainer row's
`question` or `answer`, not only in metadata.

### Flat Trainer Rows Need Rich Evidence Beside Them

The trainer needs simple rows:

```json
{"image_path":"images/example.jpg","question":"Describe this image in detail.","answer":"A grounded answer."}
```

That shape is correct for training, but too thin for review. The archive,
review file, and report preserve source labels, generated candidates, row
origins, validation results, review decisions, selected rows, rejected rows,
and readiness blockers.

### Long Jobs Need Set-And-Forget Behavior

Dataset-scale generation cannot depend on a single browser session or a single
model call behaving perfectly. During hardening, the observed and anticipated
failure modes included oversized prompts, confusing output-token traces,
repeated-token loops, full-image composition stalls, missing model files,
backend restarts, and MLX/Metal GPU failures.

The implementation therefore uses backend jobs, persisted job state, progress
polling, attach/recover behavior, active-job guards, prompt-budget telemetry,
loop detection, safe retries, runtime unload on repeated-output loops, and
process-level isolation for dataset jobs.

## Runtime And Prompt Hardening

| Risk | Implemented behavior |
| --- | --- |
| User output cap did not match runtime output cap | UI separates Auto from explicit numeric override; logs record requested and effective budgets |
| Thinking-capable models need larger defaults | Auto budgets can be higher for thinking-capable models while user numeric caps remain hard caps |
| Dense scenes made prompts too large | Counts remain authoritative; dense box context is represented by a capped spatial subset |
| Large prompts can crowd output space | Prompt previews and runtime prompt measurement adapt non-explicit output budgets |
| Full-image visual composition could stall | Windowed set-and-forget mode can compose from completed crop evidence instead of resending the full image tensor |
| Repeated punctuation looked like a hang | Streaming loop inspection detects repeated surfaces, trims progress display, raises a controlled loop error, and routes recovery |
| Repeated loop recovery can poison the runtime | Recovery can unload the model before retrying with safer decode settings |
| GPU faults can abort the worker process | Dataset runs use isolated child attempts and persisted status so process aborts become observable job failures |
| Browser refresh loses progress | Job metadata, progress, attach/recover, and default-root discovery mirrors preserve job visibility |
| Model availability is ambiguous | Model selectors visually distinguish download-needed models from locally available models |

## Archive And Concurrency Hardening

Caption and instruction jobs mutate the caption archive. The UI and backend now
treat that archive as a shared resource with explicit busy-state guards.

Implemented protections include:

- caption image, caption next-N, caption all, and training-dataset launch paths
  check archive-idle state at action time;
- caption output editing and autosave are blocked while a live job mutates the
  archive;
- prompt, model, token, decode, glossary, style, recipe, row-family, and
  instruction settings lock during active jobs;
- current-image caption archive reads defer while the archive is mutating;
- backend read, write, export, and review-import routes reject same-dataset
  operations while a caption job is active;
- review import checks busy state before opening the file picker and again
  before importing a selected file;
- export failures use shared operator-facing status formatting;
- report export validates sibling trainer, archive, and review row shapes plus
  cross-artifact consistency and instruction settings fingerprints before
  saving the run summary;
- custom output directories mirror `job.json` into the default discovery root
  so active jobs are visible after restart or auto-resume.

The goal is not just disabled buttons. Stale clicks, stale input events, direct
API calls, and persisted live runners should all fail closed around the active
caption archive.

## UI/UX Contract

An external reviewer should expect the UI to behave as follows:

- the model selector makes local availability clear;
- ordinary captioning and training-dataset creation are separate actions;
- set-and-forget is the default posture for dataset-scale work;
- instruction options are visible and explain row-family separation;
- launch is blocked if no trainable row family is enabled;
- generated QA can be review/archive-only;
- busy archive state disables relevant controls and also blocks stale actions
  at runtime;
- status messages preserve backend block reasons instead of replacing them with
  generic failure text;
- export and review-import actions report readiness and busy failures in the
  export-health area;
- final downloads are tied to the same artifact family and run report.

## Validation Layers

The workflow validates at several boundaries:

| Boundary | What is checked |
| --- | --- |
| UI settings | row-family selection, required dataset context, visible busy state, action-time guards |
| Prompt preview | prompt size, effective output budget, dense-box subset behavior, explicit cap behavior |
| Backend job request | normalized instruction settings, same-dataset active job conflicts, persisted runner conflicts |
| Archive construction | source/generated separation, label-count contradictions, structured rewrite rules, manifest membership |
| Review import | stable ids, dataset identity, image aliases, stale text, duplicate actionable rows, malformed rows |
| Artifact export | trainer rows, review rows, report counts, archive/report/review consistency, instruction settings fingerprint consistency |
| Trainer loader | flat row shape, image resolution, duplicate image/question pairs, invalid review states, required metadata |
| Soak/readiness tools | runtime trace binding, loop recovery evidence, unattended certification gates |

This is deliberately redundant. The browser catches operator mistakes early,
the backend protects script/API callers, and the trainer loader rejects unsafe
files before fine-tuning.

## Current Readiness Claim

The implementation is ready for:

- external code and artifact-contract review;
- a small real-data pilot;
- UI/UX review of the training-dataset workflow;
- trainer-loader dry-run validation on generated pilot rows.

The implementation is not yet claiming:

- every generated row is factually correct;
- every target dataset will produce training-ready output without review;
- large-scale fine-tuning should begin without a pilot;
- deterministic metadata QA should be enabled for every corpus by default;
- GPU/runtime faults are impossible.

The product claim is narrower and stronger: the workflow now exposes the right
training-data generation path, records provenance, keeps source truth separate
from generated language, guards long jobs, and provides validation boundaries
so unsafe rows can be caught before training.

## Required Pilot Before Training Use

Before a generated corpus is used for fine-tuning, run a pilot and record:

- image count;
- images with source labels;
- images with empty or missing labels;
- training bundle image checksum verification;
- `caption0` selected and rejected counts;
- generated QA candidate count;
- generated QA selected count;
- generated QA rejection reasons;
- deterministic metadata QA row count;
- manual-review-required count;
- review accepted, rejected, and needs-revision counts;
- duplicate image/question rejection count;
- invalid JSON answer count when JSON answers are requested;
- structured source rewrite count;
- artifact-consistency status;
- strict trainer export readiness status;
- trainer-loader import result;
- loop recovery rate;
- signal-exit or child-process failure rate;
- deterministic fallback rate;
- failed-attempt row rate;
- p95 runtime projection.

Only after that pilot passes the configured health, review, consistency, and
trainer-loader gates should the same settings be used for a larger corpus.

## Recommended External Review Questions

1. Does the UI expose the requested training-dataset workflow as a first-class
   action rather than a hidden caption export?
2. Are generated captions and generated QA kept separate from source labels?
3. Can a reviewer trace every flat trainer row back to image, question, answer,
   row origin, validation state, review state, and run report?
4. Do rejected or review-pending rows stay in review/audit artifacts while
   being excluded from trainer JSONL?
5. Are dense prompts bounded without losing authoritative counts?
6. Does an explicit numeric output-token setting remain a hard user override?
7. Are long-running jobs recoverable after refresh or backend restart?
8. Do stale UI clicks, direct API calls, exports, and review imports fail
   closed while the caption archive is mutating?
9. Does the trainer loader reject unsafe rows before fine-tuning starts?
10. Does the pilot produce enough accepted, diverse, source-grounded rows to
    justify scaling the generation settings?
